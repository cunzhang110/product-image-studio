# 批次复制规则与 Muzhi 默认值调整设计

## 目标

让复制产品批次成为随时可用的“复用参考图与配置、新开一轮创作”动作，并将新建批次的默认生图服务改为 Muzhi。

## 批次复制规则

复制按钮在原批次的所有运行阶段都可用，包括生成提示词、生成主场景和生成图片期间。复制动作只读取点击瞬间的批次数据，不停止、不取消也不修改原批次的任务。

新副本保留：

- 产品参考图和风格参考图。
- 提示词模板。
- 请求数量、操作模式、生成策略和同场景分支模式。
- 自定义延伸节点。
- 提示词服务及模型设置。
- 图片服务、图片模型、画幅、分辨率和批次并发设置。

新副本始终重置：

- `creativeGuide` 设为空字符串。
- `prompts` 和 `images` 设为空数组。
- `runPhase` 设为 `idle`，清空 `runError`。
- 清空 `sceneBible` 和 `anchorImageId`。
- `stage` 设为 `setup`。
- 生成新的批次 ID、延伸节点 ID、创建时间和更新时间。

名称仍使用“原名称 - 副本”“原名称 - 副本 2”的递增规则。复制不产生提示词或图片 API 请求。

## 默认生图服务

`createProductBatch()` 创建的新批次默认使用：

- `imageProvider: "muzhi"`
- `imageModel: "gpt-image-2"`

该默认值只影响之后创建的新批次，包括工作区为空时创建的初始批次。读取已有批次时不迁移供应商；复制批次保留来源批次当前选择的供应商和模型。

## UI 与持久化

- 移除复制按钮基于 `generationActive` 的禁用状态。
- `App.tsx` 继续调用领域层 `duplicateProductBatch()`，不在组件中拼装复制规则。
- 新副本插入列表顶部并自动选中。
- 现有 IndexedDB 自动保存机制继续持久化副本，不修改数据库版本。

## 测试

领域测试验证：

- 复制始终清空提示词、创作引导、图片和运行状态并进入设置页。
- 运行中的来源批次也能产生干净副本，且来源批次保持不变。
- 参考图、模板、策略、节点和供应商配置得到保留。
- 新建批次默认使用 Muzhi 和 `gpt-image-2`。
- 已有批次归一化不被强制改成 Muzhi。

React 测试验证：

- 运行期间复制按钮仍可点击。
- 点击后立即出现并选中干净副本。
- 原批次继续运行，未被取消。
- 副本通过现有自动保存流程持久化。

## 非目标

- 不提供可选择复制范围的弹窗。
- 不等待原批次完成后再复制。
- 不把已有批次或复制来源批次强制切换到 Muzhi。
- 不修改供应商调度、密钥或 API 请求格式。
