# 复制产品批次设计

## 目标

允许用户从当前产品批次创建一个可再次生图的副本，避免重新上传风格参考图、产品参考图或重复配置参数。副本保留已审核的提示词，但不保留任何旧生图任务和结果。

## 用户交互

- 在左侧产品批次栏增加“复制当前批次”按钮，位置与“新建产品批次”“删除当前批次”相邻。
- 点击后立即创建副本，不增加确认弹窗。
- 新副本插入批次列表顶部，并自动切换为当前批次。
- 当前批次正在生成时禁用复制按钮，避免复制执行过程中的中间状态。
- 复制成功后显示轻量成功提示。

## 命名规则

- 第一次复制使用 `原名称 - 副本`。
- 当该名称已存在时，依次使用 `原名称 - 副本 2`、`原名称 - 副本 3`。
- 名称冲突判断基于当前工作区的全部批次名称。
- 副本名称视为系统自动生成；用户之后仍可手动修改。

## 数据复制规则

在 `domain/productWorkflow.ts` 增加领域函数 `duplicateProductBatch(source, existingNames)`，集中维护复制语义。

副本保留：

- 风格参考图与产品参考图。
- 提示词模板与创作引导。
- 请求数量、操作模式、生成策略和同场景分支模式。
- 自定义延伸节点。
- 提示词列表、提示词文本及选中状态。
- 提示词服务与模型快照。
- 图片服务、模型、画幅、分辨率和批次并发设置。

副本重置：

- 生成新的批次 ID。
- 生成新的提示词和延伸节点对象，避免编辑副本时修改原批次的嵌套数据。
- 清空 `images`，不复制生图任务、结果、错误或请求快照。
- `runPhase` 重置为 `idle`，清空 `runError`。
- 清空 `sceneBible` 和 `anchorImageId`，避免沿用旧主场景状态。
- `stage` 设置为 `review`，让用户可以直接检查提示词并再次生图。
- `createdAt` 与 `updatedAt` 设置为复制时刻。

如果源批次没有提示词，副本的 `stage` 设置为 `setup`，避免进入空的审核页面。

## 架构与状态流

复制规则属于产品批次领域语义，因此不在 React 组件内拼装对象。`App.tsx` 只负责：

1. 确认当前批次不在运行。
2. 调用 `duplicateProductBatch()`。
3. 将副本插入批次列表顶部。
4. 切换 `activeBatchId`。
5. 显示成功提示。

现有批次自动保存机制会把更新后的列表写入 IndexedDB，不需要修改数据库结构或增加迁移。

## 错误与边界处理

- 正在生成的批次不允许复制；按钮禁用，不主动停止原批次。
- 复制不会调用提示词或图片供应商，因此不产生费用。
- 复制不会注册新的运行控制器或 Muzhi 调度任务。
- 图片数据 URL 作为不可变字符串复用；数组和对象字段创建新对象，避免引用共享。
- 不增加“复制全部结果”或“仅复制参考图”等额外模式。

## 测试

### 领域测试

在 `domain/productWorkflow.test.ts` 验证：

- 副本获得不同 ID 和正确的递增名称。
- 参考图、设置和提示词内容得到保留。
- 提示词和延伸节点为独立对象。
- 图片任务、运行错误、主场景信息和执行状态被清空。
- 有提示词时进入 `review`，无提示词时进入 `setup`。

### React 交互测试

在相邻 App 测试中验证：

- 点击复制后列表顶部出现副本并自动选中。
- 副本显示保留的提示词且没有完成图片。
- 原批次保持不变。
- 当前批次运行时复制按钮禁用。
- 批次列表变化进入现有持久化流程。

## 非目标

- 不复制已生成图片或失败任务。
- 不支持多个批次同时复制。
- 不提供复制范围选择弹窗。
- 不改变现有数据库版本、供应商请求或调度逻辑。
