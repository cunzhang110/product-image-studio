# 产品生图工作台设计说明

## 1. 目标

建立一个与现有“批量生图大师”完全独立的新项目，把用户当前分散在两个软件中的流程合并为一个连续工作流：

1. 上传一张产品参考图。
2. 填写提示词模板和创作引导。
3. 由文本模型批量生成指定数量的提示词。
4. 用户审核、修改、删除或补充提示词。
5. 将确认后的提示词批量提交给图片模型。
6. 查看、重试和批量下载生成结果。

本项目面向“同一产品的一批一致性图片”，不是生成彼此无关的创意。每个批次只使用一个产品和一张参考图。

## 2. 项目隔离

- 新产品名称：产品生图工作台。
- GitHub 仓库：`cunzhang110/product-image-studio`。
- Vercel 项目：`product-image-studio`。
- 新项目使用独立 Git 历史、独立 Vercel 项目、独立网址和独立环境变量。
- 现有仓库 `cunzhang110/piliangchutu` 和现有网站 `piliangshengtu.vercel.app` 不做任何修改。
- 初始技术底座从现有线上主分支复制，以复用已经验证过的 API 接入和生图能力。

## 3. 产品原则

### 3.1 产品批次优先

一级对象是“产品批次”，不是单条生图任务。一个批次包含：

- 产品名称。
- 一张产品参考图。
- 提示词模板。
- 创作引导。
- 提示词数量。
- 默认比例和分辨率。
- 文本模型配置。
- 生图服务配置。
- 由该批次生成的提示词和图片结果。

### 3.2 固定产品，变化画面

同一批次中的所有提示词都必须围绕同一产品。产品外观和识别特征来自同一张参考图及用户模板；场景、构图、光线、镜头、摆放方式和人物互动可以变化。

系统不增加自动产品一致性检查。产品一致性通过参考图绑定、提示词生成指令和人工审核共同控制。

### 3.3 先审核，再生图

AI 生成提示词后只进入“待审核”状态，不自动产生图片。用户确认并选择提示词后，才创建生图队列，避免错误提示词批量消耗额度。

## 4. 核心流程

### 阶段一：建立批次

用户输入：

- 批次名称或产品名称。
- 产品参考图，限一张。
- 提示词模板。
- 创作引导。
- 生成数量。
- 默认画幅和分辨率。

提示词模板用于固定必须出现的产品信息和表达框架。创作引导用于描述本批次希望变化的方向，例如生活化场景、人物手持、桌面静物、自然光或商业广告风格。

### 阶段二：生成提示词

支持图片输入的文本模型同时读取产品参考图、提示词模板、创作引导和数量，然后返回结构化提示词列表。同一张产品图会参与两个阶段：生成提示词时帮助 AI 理解产品，生成图片时作为产品参考图直接提交给图片模型。

提示词生成指令要求模型描述参考图中能够确认的产品外观和识别特征，不虚构看不到的背面、内部结构或额外配件。用户仍然可以在审核阶段自由修改这些描述，系统不做自动拦截。

文本模型与生图服务分开显示和配置，避免用户误以为生成提示词与生成图片是同一次调用。界面明确展示：

- “提示词 AI”：负责生成文字，会产生文本模型调用。
- “生图服务”：负责生成图片，会产生图片模型调用。

### 阶段三：审核提示词

生成结果以可编辑列表展示。用户可以：

- 修改单条提示词。
- 选择或取消选择。
- 删除单条提示词。
- 重新生成单条提示词。
- 重新生成全部提示词。
- 继续追加指定数量的提示词。

默认选中全部新提示词，但不会自动生图。页面始终显示已选数量。

### 阶段四：批量生图

用户点击“生成已选图片”后，系统显示最终确认信息：

- 将生成的图片数量。
- 当前生图服务和固定图片模型。
- 比例、分辨率和并发数。
- 当前绑定的产品参考图。

确认后，每条提示词变成一个独立生图任务，并自动绑定该批次的产品参考图。用户不需要在每条提示词里重复输入 `@参考图`。

### 阶段五：结果处理

生成结果与对应提示词保持关联。用户可以：

- 查看生成状态和失败原因。
- 重新生成失败任务。
- 使用原提示词再次生成。
- 修改提示词后生成新版本。
- 下载单张图片。
- 批量下载当前批次的成功图片。

## 5. 页面结构

首版采用工作台布局，不制作营销落地页。

### 左侧：批次列表

- 新建产品批次。
- 显示产品缩略图、批次名称、提示词数量、成功图片数量和更新时间。
- 切换批次时恢复该批次的完整状态。

### 中间：当前工作区

根据批次状态显示三个连续步骤：

1. 批次设置。
2. 提示词审核。
3. 生图结果。

三个步骤保留在同一批次内，用户可以返回修改，但修改批次设置不会自动覆盖已经生成的提示词和图片。

### 右侧：执行设置

- 提示词 AI 与文本模型。
- 生图服务与固定图片模型。
- 比例、分辨率和并发数。
- 本次预计调用数量。
- 当前阶段的主要执行按钮。

## 7. API 复用与边界

保留现有项目已经接入的三家生图服务：

- Yunwu：固定图片模型 `gemini-3.1-flash-image-preview`。
- APIMart：固定图片模型 `gpt-image-2`。
- Muzhi：固定图片模型 `gpt-image-2`；有参考图时使用图片编辑接口。

保留现有服务端代理、API Key 管理、限流重试、任务轮询和图片返回处理逻辑。新工作流只在调用前构造新的批次和任务数据，不重新发明第三方 API 接入。

文本模型调用单独封装，并支持把当前批次的产品参考图作为多模态输入。生成提示词时要求模型返回 JSON 数组；如果返回格式异常，系统提供可读错误和重试，不自动进入生图阶段。

## 8. 存储与密钥

- 首版继续使用浏览器本地存储保存批次、提示词和任务状态，不建立用户账号和云端数据库。
- 参考图保存在浏览器 IndexedDB，沿用现有图片压缩策略。
- API Key 继续按服务商独立保存。
- Muzhi 服务端 Key 继续通过新 Vercel 项目的环境变量配置，不写入前端代码或 Git 仓库。
- 新 Vercel 项目需要重新配置环境变量，但不需要重新开发 API。

## 9. 错误处理

- 文本模型失败：保留当前批次输入，允许重试，不创建空提示词任务。
- 文本返回数量不足：展示实际获得数量，允许追加生成缺少的数量。
- 单条生图失败：只影响该任务，不停止已经成功的任务。
- API Key 缺失或无效：明确指出是提示词 AI 还是生图服务配置问题。
- 页面刷新或关闭：从本地数据库恢复批次及任务状态。

## 10. 首版范围

首版包含：

- 产品批次管理。
- 单张产品参考图。
- 模板、引导和数量输入。
- AI 批量生成提示词。
- 提示词审核、选择、修改、删除、重生成和追加。
- 三家现有图片服务。
- 批量队列、失败重试和下载。
- 独立 GitHub 仓库与独立 Vercel 部署。

首版不包含：

- 多产品混合批次。
- 多角度参考图库。
- 自动产品一致性检查。
- 用户账号、团队协作或云端项目同步。
- 自动发布到电商或社交媒体。
- 无审核的一键提示词生成并直接生图。

## 11. 验收标准

1. 用户能在一个页面内完成“产品图 + 模板 + 引导 + 数量 → 提示词 → 图片”的完整流程。
2. 一个批次只需上传一次产品参考图，提示词 AI 和所有生图任务自动使用该图。
3. 用户可以在产生图片费用前审核和选择提示词。
4. Yunwu、APIMart 和 Muzhi 的现有生图能力继续可用。
5. 新项目的提交和部署不会触发旧项目构建，也不会改变旧网站。
6. 刷新页面后，未完成批次及其提示词仍然存在。
